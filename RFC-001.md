# RFC-001: Zinc is not an IPC library

## The original idea

I wanted fast communication between JavaScript runtimes on the same machine. Bun for the hot path, Node for the ecosystem, maybe Deno for sandboxed stuff. The usual options — Unix sockets, HTTP, WebSockets — all felt wasteful. You're on the same machine. Why are you going through the kernel's network stack?

So I built Zinc: a lock-free ring buffer in Zig, mapped into shared memory via `shm_open` + `mmap`, with FFI adapters for all three runtimes. The pitch was "zero-copy IPC at RAM speed." I wrote the Zig core, the Bun FFI bindings, the Deno plugin, a Rust N-API addon for Node, a binary serialization format, an RPC protocol layer, connection pooling, security guards. The whole stack.

It worked. Sort of.

## What was actually broken

When I really looked at what I had built, the problems were worse than expected.

**The ring was fighting itself.** Server and client shared a single ring buffer. The server would write a reply, then on its next poll tick, consume its own reply before the client ever saw it. The fundamental data structure was broken for bidirectional communication.

**"Zero-copy" was a lie.** The data path was: JS object → custom binary encoding → `memcpy` into ring slot → `memcpy` out of ring slot → binary decoding → new JS object. That's like, at minimum two userspace copies plus serialization overhead. Calling this "zero-copy" because we skipped the kernel was dishonest.

**The latency floor was JavaScript.** The Zig ring buffer could do sub-microsecond operations. But the JS layer was polling with `setInterval`, which has a minimum resolution of ~1ms in Node and ~0.5ms in Bun. The native layer was fast; the thing people actually used was slow.

**PIDs were broken on macOS.** The Zig core called `std.os.linux.getpid()`, which returns garbage on Darwin. Every message had a corrupt sender PID. The security layer's PID allowlisting was useless.

**The Node addon leaked memory.** Every `poll()` call allocated a fresh 4KB `Vec` for the payload buffer, even when the ring was empty. At the default poll rate, that's megabytes per second of allocation pressure for zero useful work.

There were more issues — missing PING/PONG handling that broke the connection pool, a serialization format that couldn't handle objects or arrays, PID fields too small for real systems — but the architectural ones were the real killers.

## The uncomfortable question

After fixing the bugs, I had to ask myself: why would anyone use this instead of a Unix socket?

The honest answer was: they probably wouldn't. The ring buffer was marginally faster for small payloads, but it had a 4KB message limit, no streaming, no backpressure, required a Zig toolchain to build, and crashed silently on errors. A Unix domain socket with JSON does all of this out of the box with zero build dependencies.

I'd built a more complex, more fragile version of something that already exists.

## The thing I was actually sitting on

Here's what I missed: the interesting part of Zinc was never the ring buffer. It was the `mmap`.

When two processes `mmap` the same shared memory segment, they're looking at the same physical pages of RAM. Not copies. Not messages. The same bytes. If process A writes `42.0` to offset 0, process B can read it at offset 0 instantly — no syscall, no copy, no serialization. It's just a memory read.

Every JS runtime can expose a raw pointer as an ArrayBuffer:

- Bun: `toArrayBuffer(ptr, 0, size)` via `bun:ffi`
- Deno: `Deno.UnsafePointerView.getArrayBuffer(ptr, size)`
- Node: `napi_create_external_arraybuffer` in the N-API addon

And once you have an ArrayBuffer, you have the entire TypedArray ecosystem: `Float32Array`, `Uint8Array`, `Int32Array`, `BigInt64Array`. You have `Atomics.wait` and `Atomics.notify` for cross-process synchronization. You have `DataView` for heterogeneous layouts.

Nobody has done this across processes in JavaScript. `SharedArrayBuffer` only works within a single process (between Worker threads). There's no cross-process equivalent. Until now.

## What Zinc actually is

Zinc is a cross-process shared memory primitive for JavaScript. Not a message queue. Not an RPC framework. A way to share raw memory between processes — across Bun, Deno, and Node — with zero copies and zero serialization.

The API is one function:

```ts
import { sharedBuffer } from "zinc";

// Process A — create a 1MB shared region
const region = await sharedBuffer("/my-data", 1024 * 1024, true);
const floats = new Float32Array(region.buffer);
floats[0] = 42.0;

// Process B — open the same region
const region = await sharedBuffer("/my-data", 1024 * 1024, false);
const floats = new Float32Array(region.buffer);
console.log(floats[0]); // 42.0 — same physical memory, zero copies
```

The ring buffer stays, but its role changes. It's the notification channel — lightweight signals like "hey, I updated offset range 0-4096" — not the data transport. Data lives in the shared region and is read directly.

## What this enables

The boring version: two processes sharing a config struct or a session cache without Redis.

The interesting version: a Bun process writing 8MB video frames at 60fps into shared memory, and a Node process reading them with zero transfer overhead. An ML pipeline where Python writes model outputs as Float32Arrays and a JS API server reads them without a network hop. A game engine where physics runs in one process and rendering in another, sharing the entire world state through typed arrays.

The performance difference isn't 2x or 5x. It's categorical. You go from "serialize, copy, deserialize" to "just read the memory." For large payloads, that's the difference between milliseconds and nanoseconds.

## What stays, what goes

**Kept:** The Zig core (extended with raw shm primitives), all three FFI adapters (extended with `SharedBuffer`), the ring buffer (repurposed as notification channel), the C ABI (enables future cross-language support).

**Added:** `sharedBuffer()` — the new core primitive. `SharedBuffer` classes in each adapter. `SharedMemoryRegion` type in the public API.

**Kept but secondary:** `serve()`/`connect()` still work for simple RPC. They're fine for small payloads where convenience matters more than performance. But they're no longer the point.

## What's next

Higher-level abstractions on top of the raw buffer: a `SharedMap` with schema-defined typed fields, a `SharedRing` for streaming typed data, and cross-language support (Python, Rust, Go) since the Zig core exports a standard C ABI.

But the foundation is the shared buffer. Everything else is sugar.
